# AVATAR OnE 기능명세서 변경사항 (v1.9 → v1.10)

> **작성일**: 2026-04-07
> **범위**: Trainer 워크로드-리소스 통합, 데드 코드 정리
> **대상**: 외주 개발업체 전달용

---

## 1. Trainer 탭 구조 변경

### 기존 (v1.9)

```
관리자 모드 Trainer: 요청 관리 | 워크로드 | 리소스 | Solver 관리 | 모델 선택
```

### 변경 (v1.10)

```
관리자 모드 Trainer: 요청 관리 | 워크로드 | Solver 관리 | 모델 선택
```

| 변경 | 내용 |
|------|------|
| "리소스" 탭 제거 | 독립 탭으로 존재하던 클러스터 리소스 현황 화면을 제거 |
| "워크로드" 탭에 통합 | 워크로드 탭 상단에 클러스터 전체 자원 현황을 표시 |

**변경 이유**: 관리자가 워크로드 실행 상태를 모니터링하면서 동시에 클러스터 자원 현황을 파악할 수 있도록 한다. 기존에는 탭을 전환해야 했으나, 통합 후에는 하나의 화면에서 모든 정보를 확인할 수 있다.

### 변경 전

워크로드 탭:
```

│ 워크로드 목록 테이블                      │
│  워크로드명 | 상태 | 우선순위 | 자원 | ... │

└─────────────────────────────────────────┘
```

워크로드 탭:

```
│  GPU V100 [██████░░░░░░] 6/8기 유휴 2    │

├─────────────────────────────────────────┤
│ (클릭 시) 학습 진행 차트 + 컴포넌트 실행  │
└─────────────────────────────────────────┘
```

### 세부 사항

| 항목 | 내용 |
|------|------|
| 자원 현황 위치 | 워크로드 테이블 상단 (2x2 그리드) |
| 표시 자원 | GPU A100, GPU V100, 메모리, CPU |
| 표시 정보 | 총량, 요청량, 사용량, 유휴량 (바 차트) |
| 데이터 소스 | 실행 중(running) + 대기(queued) 워크로드의 자원 합산 |
| 워크로드 테이블 | 기존과 동일 (변경 없음) |
| 컴포넌트별 실행 현황 | 기존과 동일 — 워크로드 클릭 시 표시 (변경 없음) |

## 4. 컴포넌트 실행 실시간 시뮬레이션

### 변경 전

- 컴포넌트 실행 진행률이 `0.6` (60%)으로 하드코딩
- 컴포넌트 목록이 9개 리터럴 배열로 고정 (spec 데이터 미참조)
- 자원 사용량(memUsed, usage)이 고정값

### 변경 후

- **실시간 진행률**: 워크로드 실행 10초 동안 컴포넌트가 순차적으로 waiting → running → completed 전환
- **동적 컴포넌트 목록**: spec의 `tasks[0].components`에서 파생, 컴포넌트 라이브러리에서 type 정보 참조
- **자원 사용량 시뮬레이션**: 실행 중인 컴포넌트만 자원 소비 표시, 대기 중인 컴포넌트는 0

### 세부 사항

| 항목 | 내용 |
|------|------|
| 진행 방식 | `setInterval` (500ms)로 0.5초마다 진행률 업데이트 |
| 기준 시간 | 워크로드의 `startedAt` 타임스탬프 기반 (실행 시작 시점부터 경과 시간 계산) |
| 컴포넌트 순서 | spec 데이터의 `order` 필드 기준 순차 실행 |
| 타입 표시 | 컴포넌트 라이브러리(`libraryRef`)에서 type(component/solver/environment/train) 참조 |
| 자원 표시 | running: GPU/메모리 사용량 표시, waiting: "—", completed: 최종 사용량 유지 |
| 상태 동기화 | `selectedWLId` + derive 패턴으로 워크로드 상태 변경 즉시 반영 |

## 5. 워크로드 이름 고유화

### 변경 전

동일 spec에서 워크로드를 여러 번 제출하면 모두 같은 이름 (예: "RCP-BladeOpt-training")

### 변경 후

동일 spec의 기존 워크로드 수를 세어 자동 접미사 추가:
- 첫 번째: `RCP-BladeOpt-training`
- 두 번째: `RCP-BladeOpt-training-2`
- 세 번째: `RCP-BladeOpt-training-3`

## 6. 컴포넌트 자원 표시 개선 (GPU/Mem/CPU 3중 바)

### 변경 전

- 단일 라벨 없는 바 하나 (퍼센트 의미 불명확)
- GPU 활용률 없음 (할당만 표시)
- CPU 사용률 완전 누락
- 자막 텍스트가 할당과 사용량 혼합 ("V100 x 1 · 12.4 / 16GB")

### 변경 후

각 컴포넌트 카드에 **GPU / Mem / CPU 3개 미니 바** 표시:

```
┌────────────────────────────────────┐
│ 1. rcp-setup          [component]  │
│ V100 x 1 · 16GB · 8코어            │  ← 할당(고정)
│ GPU  ████████████░░░░░  72%        │  ← 활용률
│ Mem  ██████████████░░░  82%        │     (동적)
│ CPU  █████░░░░░░░░░░░░  31%        │
│ ✓ 완료                              │
└────────────────────────────────────┘
```

### 세부 사항

| 항목 | 내용 |
|------|------|
| 미니 바 | GPU(파랑/보라), Mem(앰버), CPU(녹색) — RESOURCES 상수 색상 재사용 |
| 자막 텍스트 | 할당 정보만: `V100 x 1 · 16GB · 8코어` |
| GPU 바 색상 | A100=#1D4ED8(파랑), V100=#7E22CE(보라) |
| 상태별 표시 | running: 3개 바 활성, completed: 최종값, waiting: "자원 미할당" |
| 워크로드 요약 | 컴포넌트 그리드 위에 총 GPU/메모리/CPU + 진행률 표시 |
| CPU 코어 수 | `gpu_count × 4`로 시뮬레이션 |

# Change Summary: v1.9 → v1.10 (English)

> **Date**: 2026-04-07
> **Scope**: Trainer workload-resource consolidation, dead code cleanup, component resource visualization

---

## 1. Trainer Tab Structure Change

### Before (v1.9)

```
Admin Trainer: Request Mgmt | Workloads | Resources | Solver Mgmt | Model Selection
```

### After (v1.10)

```
Admin Trainer: Request Mgmt | Workloads | Solver Mgmt | Model Selection
```

The standalone "Resources" tab is removed. Its content (cluster resource utilization bars) is now displayed at the top of the "Workloads" tab.

**Reason**: Administrators can now monitor workload execution and cluster resource utilization in a single view without switching tabs.

---

## 2. Workload View — Integrated Resource Overview

The workload tab now shows cluster-wide resource utilization (GPU A100, GPU V100, Memory, CPU) in a 2x2 bar chart grid above the workload table.

| Item | Detail |
|------|--------|
| Position | Top of workload tab, above the workload table |
| Resources shown | GPU A100, GPU V100, Memory, CPU |
| Data displayed | Total, requested, used, idle (bar chart) |
| Data source | Sum of running + queued workloads' resource allocations |
| Workload table | Unchanged |
| Per-component execution status | Unchanged — shown on workload click |

## 4. Real-time Component Execution Simulation

### Before
- Component execution progress hardcoded at 60%
- Component list was a static 9-element literal array (not derived from spec data)
- Resource usage (memUsed, usage) were fixed constants

### After
- **Real-time progress**: During 10-second workload execution, components transition sequentially through waiting → running → completed
- **Dynamic component list**: Derived from spec's `tasks[0].components`, with type info from component library
- **Simulated resource usage**: Only actively running components show resource consumption; waiting components show 0

| Item | Detail |
|------|--------|
| Update interval | `setInterval` at 500ms |
| Time base | Workload's `startedAt` timestamp (elapsed since execution start) |
| Component order | Sequential based on spec's `order` field |
| Type display | Looked up from component library via `libraryRef` |
| Resource display | running: GPU/memory usage shown, waiting: "—", completed: final values |
| State sync | `selectedWLId` + derive pattern ensures workload status changes are reflected immediately |

---

## 5. Unique Workload Names

### Before
Submitting multiple workloads from the same spec produced identical names (e.g., "RCP-BladeOpt-training")

### After
Automatic counter suffix based on existing workloads with the same specId:
- First: `RCP-BladeOpt-training`
- Second: `RCP-BladeOpt-training-2`
- Third: `RCP-BladeOpt-training-3`

---

## 6. Triple Resource Bars (GPU/Mem/CPU)

### Before
- Single unlabeled progress bar per component card (meaning unclear)
- GPU utilization not shown (only allocation like "V100 x 1")
- CPU usage completely absent
- Subtitle mixed allocation with runtime values ("V100 x 1 · 12.4 / 16GB")

### After
Each component card shows **3 labeled micro-bars**: GPU, Memory, CPU

| Item | Detail |
|------|--------|
| GPU bar | Blue (#1D4ED8 for A100, #7E22CE for V100) |
| Mem bar | Amber (#B45309) |
| CPU bar | Green (#047857) |
| Subtitle | Allocation only: "V100 x 1 · 16GB · 8코어" |
| Waiting state | Bars hidden, shows "자원 미할당" |
| Workload summary | Total GPU/Mem/CPU + progress count above component grid |
| CPU cores | Simulated as `gpu_count × 4` |
