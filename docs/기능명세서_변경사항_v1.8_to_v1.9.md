# AVATAR OnE 기능명세서 변경사항 (v1.8 → v1.9)

> **작성일**: 2026-03-28
> **범위**: 메뉴 구조, 역할 분리, Builder/Trainer/Operator 재정의, UI 변경
> **대상**: 외주 개발업체 전달용

---

## 1. 메뉴 구조 변경

### 기존

```
사용자 모드: 홈 | Builder | 컴포넌트 라이브러리 | Trainer | 테스트 실행 | 워크로드 목록 | 결과 모델
관리자 모드: 홈 | 승인 관리 | 대기열 관리 | 리소스 | 워크로드 목록 | 결과 모델
```

### 변경

```
사용자: Builder | Operator
관리자: Trainer
```

| 메뉴 | 역할 | 포함 기능 |
|------|------|----------|
| **Builder** (사용자) | 앱 개발 + 학습 요청 | 글로벌 컴포넌트 라이브러리, 앱 설계(컴포넌트 배치 + 연결), 코드 구현(VS Code 연동), 학습 요청 제출 |
| **Trainer** (관리자) | 학습 실행 관리 | 요청 앱 실행 허용/반려, 워크로드 목록 및 상태 관리, 리소스 관리, Solver 컴포넌트 관리 |
| **Operator** (사용자) | 설계 탐색 | 관리자가 선택한 최종 모델로 설계 탐색, 파레토 프론트 시각화, 설계 선택 |

### 역할별 접근 범위

| 기능 | 사용자 | 관리자 |
|------|--------|--------|
| 컴포넌트 라이브러리 (일반) | ✅ Builder 내에서 | ✅ |
| Solver 컴포넌트 관리 | ❌ | ✅ Trainer에서 관리 |
| 앱 설계 | ✅ Builder | ❌ |
| 코드 구현 (VS Code) | ✅ Builder | ❌ |
| 학습 요청 제출 | ✅ Builder | ❌ |
| 요청 허용/반려 | ❌ | ✅ Trainer |
| 워크로드 목록/모니터링 | ❌ | ✅ Trainer |
| 실험 차트/비교 | ❌ | ✅ Trainer |
| 모델 최종 선택 | ❌ | ✅ Trainer |
| 설계 탐색 (Operator) | ✅ Operator | ❌ |

---

## 2. Builder 변경

### 2.1 글로벌 컴포넌트 라이브러리를 Builder 내로 이동

기존에 별도 메뉴였던 "컴포넌트 라이브러리"가 Builder 내부에 포함된다. Solver 타입 컴포넌트는 관리자(Trainer)에서만 관리한다.

### 2.2 컴포넌트 연결 시 메서드 입력 방식 변경

| 기존 | 변경 |
|------|------|
| 소스 메서드: 드롭다운 선택 | **소스/대상 메서드 모두 직접 입력** |
| 대상 메서드: 드롭다운 선택 | 사용자가 연결하면서 메서드명을 타이핑 |

### 2.3 코드 구현: VS Code 연동

| 기존 | 변경 |
|------|------|
| 앱 내 textarea 에디터 | **온라인 VS Code 화면으로 이동** |

- 컴포넌트 설계 완료 후 "코드 구현" 탭 클릭 → VS Code 화면으로 이동
- 우선은 `https://github.dev/ipython/ipython` 링크로 연결
- 최종 목적: 앱의 base.py, rcp_app.py, 구현체 코드가 하나의 디렉토리에 있고, VS Code에서 해당 디렉토리를 편집. 앱의 수정사항과 Volume이 동기화

### 2.4 학습 요청을 Builder에서 직접 제출

기존에 별도 화면이었던 Trainer(학습 요청)가 Builder의 워크플로우 안으로 통합된다. Builder에서 앱 설계 → 코드 구현 → 학습 요청 제출까지 한 흐름으로 진행.

---

## 3. Trainer 변경 (관리자 전용)

### 3.1 역할 재정의

기존 Trainer = "학습 요청 제출" → 변경: **관리자가 요청을 관리하는 영역**

포함 기능:
- 학습 요청 허용/반려 (승인)
- 워크로드 목록 및 실행 상태 모니터링
- 학습 차트 (reward, efficiency, critic loss, Q-value)
- 실험 비교 (같은 App 워크로드 간)
- 리소스 관리
- **Solver 컴포넌트 관리** (관리자만)
- **모델 최종 선택** (Operator로 넘길 모델 지정)

### 3.2 Solver 컴포넌트 관리

Solver 타입 컴포넌트(MakeBlade, MeshGenerator, RunCFX 등)는 관리자가 Trainer 영역에서 관리한다. 일반 사용자는 Builder에서 Solver를 사용할 수 있지만 등록/수정은 불가.

### 3.3 모델 최종 선택

워크로드 완료 후 생성된 모델 중, 관리자가 최종 선택한 모델만 Operator에서 사용 가능하게 된다.

---

## 4. Operator 변경

### 4.1 모델 소스

기존: 모든 완료된 모델을 Operator에서 사용
변경: **관리자가 Trainer에서 최종 선택한 모델만** Operator에서 사용 가능

---

## 5. 워크로드 실행 — 프론트엔드 처리

실제로는 하나의 Docker 이미지가 하나의 Pod에서 실행되지만, **데모를 위해 프론트엔드에서 컴포넌트별로 다른 자원 할당과 사용량이 표시**되어야 한다.

| 실제 | 프론트엔드 표시 |
|------|---------------|
| 1개 Pod, 1개 이미지 | 컴포넌트별 개별 GPU/메모리 할당 표시 |
| 전체 자원을 공유 | 컴포넌트별 사용량 분리 표시 |

---

## 6. 글로벌 컴포넌트 라이브러리 표시 변경

### 6.1 Builder App 목록 하단에 표시

Builder 진입 시 App 목록 하단에 글로벌 컴포넌트 라이브러리가 표시된다. **Solver 포함 전체 컴포넌트**가 표시된다.

| 기존 | 변경 |
|------|------|
| Solver 제외하고 표시 | **Solver 포함 전체 표시** |
| App 관리 탭에서만 표시 | **App 목록 화면 하단에도 표시** |

### 6.2 Solver 관리 권한

- 사용자: 라이브러리에서 Solver를 **조회**하고 Builder에서 **사용** 가능
- 관리자: Trainer에서 Solver를 **등록/수정/삭제** 가능

---

## 7. 자원 임계치 삭제

자원 임계치(GPU ≥ 4기, 메모리 ≥ 128GB → 관리자 승인 필요) 기능을 삭제한다. 사이드바의 임계치 표시도 제거된다.

| 기존 | 변경 |
|------|------|
| 임계치 초과 시 관리자 승인 필요 | **삭제** — 모든 요청은 관리자가 Trainer에서 관리 |
| 사이드바에 임계치 표시 | **삭제** |

---

## 8. 구현 요약

| # | 변경 | 상세 |
|---|------|------|
| 1 | 메뉴 구조 | 사용자(Builder, Operator) + 관리자(Trainer) |
| 2 | 컴포넌트 라이브러리 | Builder App 목록 하단에 표시. Solver 포함 전체 |
| 3 | Solver 관리 | 관리자만 등록/수정/삭제. 사용자는 조회/사용만 |

| 5 | 코드 구현 | textarea 제거, VS Code 링크로 이동 |
| 6 | 학습 요청 | Builder에서 직접 제출 |
| 7 | Trainer | 관리자 전용 (승인 + 워크로드 + 차트 + 모델 선택) |

# AVATAR OnE Changes (v1.8 → v1.9) — For Outsource Team

> **Date**: 2026-03-28
> **Scope**: Menu restructure, role separation, Builder/Trainer/Operator redefinition, UI changes

---

## 1. Menu Structure Change

### Before

```
User mode: Home | Builder | Component Library | Trainer | Test Run | Workloads | Models
Admin mode: Home | Approval | Queue | Resources | Workloads | Models
```

### After

```
User: Builder | Operator
Admin: Trainer
```

| Menu | Role | Features |
|------|------|----------|
| **Builder** (User) | App development + training request | Global component library, app design (component placement + connections), code implementation (VS Code integration), submit training request |
| **Trainer** (Admin) | Training execution management | Approve/reject app execution requests, workload list & status monitoring, resource management, Solver component management, model final selection |
| **Operator** (User) | Design exploration | Explore optimal designs using admin-selected models, Pareto front visualization, design selection |

### Role Access

| Feature | User | Admin |
|---------|------|-------|
| Component Library (general) | ✅ Inside Builder | ✅ |
| Solver Component Management | ❌ | ✅ In Trainer |
| App Design | ✅ Builder | ❌ |
| Code Implementation (VS Code) | ✅ Builder | ❌ |
| Submit Training Request | ✅ Builder | ❌ |
| Approve/Reject Requests | ❌ | ✅ Trainer |
| Workload List/Monitoring | ❌ | ✅ Trainer |
| Training Charts/Comparison | ❌ | ✅ Trainer |
| Model Final Selection | ❌ | ✅ Trainer |
| Design Exploration (Operator) | ✅ Operator | ❌ |

---

## 2. Builder Changes

### 2.1 Component Library Moved Inside Builder

The "Component Library" that was a separate menu is now inside Builder. Solver-type components are managed only by admin (in Trainer).

### 2.2 Method Input: Free Text Instead of Dropdown

| Before | After |
|--------|-------|
| Source method: dropdown selection | **Both source and target methods: free text input** |
| Target method: dropdown selection | User types method names while connecting |

### 2.3 Code Implementation: VS Code Integration

| Before | After |
|--------|-------|
| In-app textarea editor | **Navigate to online VS Code** |

- After component design, clicking "Code Implementation" tab → opens VS Code
- For now, link to `https://github.dev/ipython/ipython`
- Final goal: app's base.py, rcp_app.py, and implementation code are in one directory. VS Code edits that directory. App modifications and Volume are synced.

### 2.4 Training Request Submitted From Builder

The training request submission (previously in Trainer screen) is now part of the Builder workflow. Users go through: app design → code implementation → submit training request — all in Builder.

---

## 3. Trainer Changes (Admin Only)

### 3.1 Role Redefinition

Before: Trainer = "submit training request"
After: **Admin manages all training-related operations**

Includes:
- Approve/reject training requests
- Workload list and execution monitoring
- Training charts (reward, efficiency, critic loss, Q-value)
- Experiment comparison (same-App workloads)
- Resource management
- **Solver component management** (admin only)
- **Model final selection** (select which models go to Operator)

### 3.2 Solver Component Management

Solver-type components (MakeBlade, MeshGenerator, RunCFX, etc.) are managed by admin in Trainer area. Regular users can use Solvers in Builder but cannot register/modify them.

### 3.3 Model Final Selection

After workload completion, admin selects the final model in Trainer. Only admin-selected models become available in Operator.

---

## 4. Operator Changes

### 4.1 Model Source

Before: All completed models available in Operator
After: **Only admin-selected models from Trainer** are available in Operator

---

## 5. Workload Display — Frontend Only

In reality, one Docker image runs in one Pod. But **for the demo, the frontend should display different resource allocation and usage per component**.

| Reality | Frontend Display |
|---------|-----------------|
| 1 Pod, 1 image | Per-component GPU/memory allocation shown |
| Resources shared | Per-component usage displayed separately |

---

## 6. Component Library Display Change

### 6.1 Shown at Builder App List

The global component library is displayed below the App list on Builder's main screen. **All components including Solvers** are shown.

| Before | After |
|--------|-------|
| Solvers excluded | **All components including Solvers shown** |
| Only in App management tab | **Also shown on App list screen** |

### 6.2 Solver Management Permissions

- User: Can **view** Solvers in library and **use** them in Builder
- Admin: Can **register/edit/delete** Solvers in Trainer

---

## 7. Resource Threshold Removed

The resource threshold feature (GPU ≥ 4 or Memory ≥ 128GB → requires admin approval) is removed. The sidebar threshold display is also removed.

| Before | After |
|--------|-------|
| Threshold exceeded → admin approval required | **Removed** — all requests managed by admin in Trainer |
| Threshold shown in sidebar | **Removed** |

---

## 8. Implementation Summary

| # | Change | Detail |
|---|--------|--------|
| 1 | Menu structure | User (Builder, Operator) + Admin (Trainer) |
| 2 | Component library | Shown on Builder App list. All components including Solvers |
| 3 | Solver management | Admin only for register/edit/delete. Users can view/use |
| 4 | Method input | Remove dropdowns, free text typing |
| 5 | Code implementation | Remove textarea, link to VS Code |
| 6 | Training request | Submit from Builder directly |
