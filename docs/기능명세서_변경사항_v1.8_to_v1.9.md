# AVATAR OnE 기능명세서 변경사항 (v1.8 → v1.9)

> **작성일**: 2026-03-28
> **범위**: 메뉴 구조, 역할 분리, Builder/Trainer/Operator 재정의, UI 변경
> **대상**: 외주 개발업체 전달용

---

## 1. 메뉴 구조 변경

### 기존

```
사용자 모드: 홈 | Builder | 컴포넌트 라이브러리 | Trainer | 테스트 실행 | 워크로드 목록 | 결과 모델
관리자 모드: 홈 | 승인 관리 | 대기열 관리 | 리소스 | 워크로드 목록 | 결과 모델
```

### 변경

```
사용자: Builder | Operator
관리자: Trainer
```

| 메뉴 | 역할 | 포함 기능 |
|------|------|----------|
| **Builder** (사용자) | 앱 개발 + 학습 요청 | 글로벌 컴포넌트 라이브러리, 앱 설계(컴포넌트 배치 + 연결), 코드 구현(VS Code 연동), 학습 요청 제출 |
| **Trainer** (관리자) | 학습 실행 관리 | 요청 앱 실행 허용/반려, 워크로드 목록 및 상태 관리, 리소스 관리, Solver 컴포넌트 관리 |
| **Operator** (사용자) | 설계 탐색 | 관리자가 선택한 최종 모델로 설계 탐색, 파레토 프론트 시각화, 설계 선택 |

### 역할별 접근 범위

| 기능 | 사용자 | 관리자 |
|------|--------|--------|
| 컴포넌트 라이브러리 (일반) | ✅ Builder 내에서 | ✅ |
| Solver 컴포넌트 관리 | ❌ | ✅ Trainer에서 관리 |
| 앱 설계 | ✅ Builder | ❌ |
| 코드 구현 (VS Code) | ✅ Builder | ❌ |
| 학습 요청 제출 | ✅ Builder | ❌ |
| 요청 허용/반려 | ❌ | ✅ Trainer |
| 워크로드 목록/모니터링 | ❌ | ✅ Trainer |
| 실험 차트/비교 | ❌ | ✅ Trainer |
| 모델 최종 선택 | ❌ | ✅ Trainer |
| 설계 탐색 (Operator) | ✅ Operator | ❌ |

## 2. Builder 변경

### 2.1 글로벌 컴포넌트 라이브러리를 Builder 내로 이동

기존에 별도 메뉴였던 "컴포넌트 라이브러리"가 Builder 내부에 포함된다. Solver 타입 컴포넌트는 관리자(Trainer)에서만 관리한다.

### 2.2 컴포넌트 연결 시 메서드 입력 방식 변경

| 기존 | 변경 |
|------|------|
| 소스 메서드: 드롭다운 선택 | **소스/대상 메서드 모두 직접 입력** |
| 대상 메서드: 드롭다운 선택 | 사용자가 연결하면서 메서드명을 타이핑 |

### 2.3 코드 구현: VS Code 연동

| 기존 | 변경 |
|------|------|
| 앱 내 textarea 에디터 | **온라인 VS Code 웹 화면으로 이동** |

- 컴포넌트 설계 완료 후 "코드 구현" 탭 클릭 → "VS Code에서 코드 작성" 버튼 표시
- 버튼 클릭 시 VS Code 웹 에디터(code-server 등)가 새 탭으로 열림
- 우선은 `https://github.dev/ipython/ipython` 링크로 연결 (데모용)

#### VS Code 웹 ↔ Volume 동기화 (구현 목표)

VS Code 웹 에디터에 마운트된 Volume에는 **컴포넌트 설계 시 자동 생성된 코드가 미리 포함**되어 있어야 한다:

```
/workspace/{app-id}/          ← VS Code에 마운트되는 Volume
├── base.py                   ← 플랫폼 프레임워크 (고정, 자동 배치)
├── {app}_app.py              ← Builder 컴포넌트 설계에서 자동 생성된 앱별 Base Class
├── base_class_spec.json      ← Builder에서 생성한 클래스 구조 정의 JSON
├── requirements.txt          ← 의존성 (자동 배치)
└── (사용자가 여기에 구현체 파일을 작성)
    ├── rcp_setup.py           ← RCPSetupBase를 상속받아 구현
    ├── impeller_agent.py      ← ImpellerAgentBase를 상속받아 구현
    ├── train.py               ← TrainBase를 상속받아 구현
    ├── run.py                 ← 실행 진입점
    └── ...
```

**흐름**:
1. 사용자가 Builder "컴포넌트 설계" 탭에서 컴포넌트 배치 + 연결 완료
2. "Base Class 생성" 클릭 → `{app}_app.py`가 생성됨
3. 생성된 `base.py` + `{app}_app.py` + `base_class_spec.json`이 **해당 App의 Volume에 자동 저장**됨
4. 사용자가 "코드 구현" 탭 → "VS Code에서 코드 작성" 버튼 클릭
5. VS Code 웹이 열리며, 해당 Volume이 마운트된 상태 → 자동 생성된 파일들이 이미 존재
6. 사용자는 VS Code에서 구현체 파일(rcp_setup.py 등)을 작성
7. 사용자가 작성한 코드는 Volume에 실시간 저장 → 학습 요청 제출 시 이 Volume의 코드가 실행됨

**백엔드 구현 시 참고**:
- App별로 PVC(Persistent Volume Claim)를 생성하여 코드를 저장
- VS Code 웹(code-server)은 해당 PVC를 마운트하여 실행
- 학습 워크로드 Pod도 동일한 PVC를 마운트하여 코드를 실행
- 컴포넌트 설계에서 Base Class 생성 시 API를 통해 PVC에 파일 쓰기

### 2.4 학습 요청을 Builder에서 직접 제출

기존에 별도 화면이었던 Trainer(학습 요청)가 Builder의 워크플로우 안으로 통합된다. Builder에서 앱 설계 → 코드 구현 → 학습 요청 제출까지 한 흐름으로 진행.

### 2.5 사전 테스트 필수화 + 자동 첨부

| 기존 (v1.8) | 변경 (v1.9) |
|------------|------------|
| 사전 테스트 선택사항 | **사전 테스트 필수** |
| 테스트 결과 첨부 또는 미첨부 선택 | **테스트 결과 자동 첨부** (미첨부 옵션 제거) |
| 테스트 없이 제출 가능 | **테스트 통과 전 제출 버튼 비활성화** |

**학습 요청 흐름**:
1. App 선택 + 학습 파라미터/자원 설정
2. **사전 테스트 실행 (필수)** → 통과 확인
3. "학습 요청 제출 (테스트 결과 첨부)" 클릭 → 테스트 결과 자동 첨부
4. 관리자 Trainer에서 **테스트 결과 확인** → 승인

**관리자 승인 흐름**:
1. Trainer "요청 관리" 탭에서 대기 중인 요청 확인
2. 첨부된 테스트 결과 확인 (수렴 여부, 보상 추이, 로그)
3. 승인 클릭 → 실행 대기열 진입

## 3. Trainer 변경 (관리자 전용)

### 3.1 역할 재정의

기존 Trainer = "학습 요청 제출" → 변경: **관리자가 요청을 관리하는 영역**

포함 기능:
- 학습 요청 허용/반려 (승인)
- 워크로드 목록 및 실행 상태 모니터링
- 학습 차트 (reward, efficiency, critic loss, Q-value)
- 실험 비교 (같은 App 워크로드 간)
- 리소스 관리
- **Solver 컴포넌트 관리** (관리자만)
- **모델 최종 선택** (Operator로 넘길 모델 지정)

### 3.2 워크로드 상세 — 학습 차트 + 실험 비교

워크로드 목록에서 행 클릭 시 상세 패널이 표시된다:

**[학습 진행] 탭**:
- 4개 차트 (2x2 그리드): Reward, Efficiency (83.9% baseline 표시), Critic Loss, Predict Q-value
- 각 차트: 원본 데이터(반투명) + 이동평균 오버레이
- 요약 바: Best Efficiency, Step, Latest Reward
- 컴포넌트별 실행 현황 (실행 순서 + 자원 사용량)

**[실험 비교] 탭**:
- 같은 App의 워크로드를 체크박스로 선택
- Efficiency/Reward 오버레이 차트 (색상별 구분 + 범례)
- 하이퍼파라미터 비교 테이블 (actor_lr, critic_lr, minibatch_size, warm_up_num, random_seed, impeller/diffuser_variation_ratio, blade_range, Best Efficiency)
- 차이가 있는 파라미터 빨간색 강조

### 3.3 Solver 컴포넌트 관리

Solver 타입 컴포넌트(MakeBlade, MeshGenerator, RunCFX 등)는 관리자가 Trainer 영역에서 관리한다. 일반 사용자는 Builder에서 Solver를 사용할 수 있지만 등록/수정은 불가.

### 3.4 모델 최종 선택 + 비교

워크로드 완료 후 생성된 모델을 관리한다:

**모델 목록 테이블**:
- App명, 워크로드명, 크기, 성능, 생성일시, Operator 배포 토글
- 행 클릭 시 상세 패널 표시

**모델 상세 패널**:
- 성능 요약 카드: Best Efficiency, Head Coefficient, TPLC, Total Steps
- 추가 지표: Flow Coefficient, Axial Thrust, Model Size
- 원본 정보: 어떤 App/워크로드에서 생성되었는지 표시

**모델 비교**:
- 같은 App의 모델을 체크박스로 선택
- 지표별 비교 테이블 (최적값 녹색 강조)
- Best Efficiency, Head Coefficient, TPLC, Axial Thrust 등

**Operator 배포**:
- 토글 스위치로 ON/OFF
- ON으로 설정된 모델만 사용자 모드 Operator에 표시

## 5. 워크로드 실행 — 프론트엔드 처리 (데모용)

> **참고**: 이 기능은 데모 시연 목적이며 실제 자원 사용량을 측정하여 보여주는 것이 아닙니다. 프론트엔드에서 시뮬레이션된 값을 표시합니다. 실제 자원 모니터링은 향후 Prometheus + DCGM 연동 시 구현됩니다.

실제로는 하나의 Docker 이미지가 하나의 Pod에서 실행되지만, **데모를 위해 프론트엔드에서 컴포넌트별로 다른 자원 할당과 사용량이 표시**되어야 한다.

| 실제 | 프론트엔드 표시 |
|------|---------------|
| 1개 Pod, 1개 이미지 | 컴포넌트별 개별 GPU/메모리 할당 표시 |
| 전체 자원을 공유 | 컴포넌트별 사용량 분리 표시 |

### 5.1 컴포넌트별 실행 현황 상세

관리자 Trainer → 워크로드 탭에서 워크로드 선택 시 컴포넌트별 실행 현황이 표시된다:

**실행 순서 흐름도**: 각 컴포넌트가 워크플로우 순서에 맞춰 왼→오른쪽으로 나열된다. 완료(녹색)/실행 중(파란색)/대기(회색)으로 상태를 구분한다.

```
1.rcp-setup ✓ → 2.impeller-agent ✓ → 3.diffuser-agent ✓ → 4.multi-agent ● → ... → 9.train (대기)
```

**컴포넌트 자원 카드 (3열 그리드)**: 각 컴포넌트별로 다음을 표시:
- 실행 순서 번호 + 이름 + 타입 배지 (component/solver/environment/train)
- 할당 GPU + 메모리 사용량 (예: V100 x 1 · 12.3 / 16 GB)
- 사용률 프로그레스 바 (녹색: 정상, 빨간색: 80% 초과, 파란색: 실행 중)
- 상태 (✓ 완료 / ● 실행 중 / 대기)
- 대기 상태 컴포넌트는 반투명 처리

9개 컴포넌트: rcp-setup → impeller-agent → diffuser-agent → multi-agent → make-blade → mesh-generator → run-cfx → environment → train

### 5.2 컴포넌트별 자원 모니터링 (실시간)

실제로는 단일 컨테이너(1 Pod, 1 이미지)에서 실행되지만, 프론트엔드에서는 **컴포넌트별로 독립된 자원이 할당되어 사용되는 것처럼** 표시해야 한다.

#### 표시 항목 (컴포넌트별)

| 항목 | 설명 | 표시 형식 |
|------|------|----------|
| CPU | 할당량 대비 사용량 | 프로그레스 바 + 수치 (예: 2.4 / 4 cores) |
| Memory | 할당량 대비 사용량 | 프로그레스 바 + 수치 (예: 12.3 / 16 GB) |
| GPU | 할당 GPU 타입/수 + 사용률 | 프로그레스 바 + 수치 (예: V100 x 1, 73%) |

#### 실시간 변화 시나리오

워크로드가 실행 중일 때, 각 컴포넌트의 자원 사용량이 **시간에 따라 변화**한다:

```
시간 흐름 →

1. rcp-setup (실행 중)
   CPU ████████░░ 78%    → (완료되면 0%로 감소)
   MEM ██████░░░░ 58%    → (완료되면 반환)
   GPU V100 x 1, 45%     → (완료되면 반환)

2. impeller-agent (실행 중)
   CPU ██████████ 92%    → (상태 정규화 연산 집중)
   MEM ████░░░░░░ 38%
   GPU V100 x 1, 61%

   ...

7. run-cfx (실행 중)
   CPU ██████████ 95%    → (CFX 시뮬레이션, CPU 집중)
   MEM ████████░░ 75%
   GPU A100 x 2, 88%     → (GPU 집중 사용)

9. train (실행 중)
   CPU ████████░░ 72%
   MEM ████████████ 98%  → (DRL 네트워크 + 경험 버퍼)
   GPU A100 x 4, 85%     → (Actor/Critic 네트워크 학습)
```

#### 구현 방식

**프론트엔드 시뮬레이션**: 백엔드에서 실제 자원 사용량을 받는 것이 아니라, 프론트엔드에서 각 컴포넌트 타입별 특성에 맞는 사용 패턴을 시뮬레이션한다.

| 컴포넌트 타입 | CPU 패턴 | Memory 패턴 | GPU 패턴 |
|-------------|---------|------------|---------|
| component (rcp-setup, agent) | 중간 (50~80%) | 낮음~중간 (30~60%) | 낮음 (20~50%) |
| solver (make-blade, mesh, cfx) | 높음 (80~95%) | 중간~높음 (60~80%) | 높음 (70~90%) |
| environment | 중간 (40~60%) | 높음 (70~85%) | 중간 (50~70%) |
| train | 중간~높음 (60~80%) | 매우 높음 (85~98%) | 높음 (75~90%) |

실행 중인 컴포넌트는 해당 패턴 범위 내에서 **주기적으로 변동하는 값**을 표시한다 (5초 간격 갱신). 완료된 컴포넌트는 사용량이 0%로 표시되거나, 최종 사용량이 고정된다.

#### VS Code Web ↔ Volume Sync (Implementation Goal)

The Volume mounted in VS Code web must **already contain the auto-generated code from component design**:

```
/workspace/{app-id}/          ← Volume mounted in VS Code
├── base.py                   ← Platform framework (fixed, auto-placed)
├── {app}_app.py              ← Auto-generated from Builder component design
├── base_class_spec.json      ← Class structure JSON from Builder
├── requirements.txt          ← Dependencies (auto-placed)
└── (user writes implementation files here)
    ├── rcp_setup.py           ← Implements RCPSetupBase
    ├── impeller_agent.py      ← Implements ImpellerAgentBase
    ├── train.py               ← Implements TrainBase
    ├── run.py                 ← Entry point
    └── ...
```

**Flow**:
1. User completes component placement + connections in Builder "컴포넌트 설계" tab
2. Clicks "Base Class 생성" → `{app}_app.py` is generated
3. Generated `base.py` + `{app}_app.py` + `base_class_spec.json` are **automatically saved to the App's Volume**
4. User clicks "VS Code에서 코드 작성" button in "코드 구현" tab
5. VS Code web opens with the Volume mounted → auto-generated files already present
6. User writes implementation files (rcp_setup.py, etc.) in VS Code
7. Code is saved to Volume in real-time → when training request is submitted, this Volume's code is executed

**Backend implementation notes**:
- Create a PVC (Persistent Volume Claim) per App to store code
- VS Code web (code-server) mounts the App's PVC
- Training workload Pod also mounts the same PVC to execute the code
- When Base Class is generated in component design, write files to PVC via API

## 5. Workload Display — Frontend Only

In reality, one Docker image runs in one Pod. But **for the demo, the frontend should display different resource allocation and usage per component**.

| Reality | Frontend Display |
|---------|-----------------|
| 1 Pod, 1 image | Per-component GPU/memory allocation shown |
| Resources shared | Per-component usage displayed separately |

### 5.1 Per-Component Execution Status (Detail)

When admin selects a workload in Trainer → Workloads tab, per-component execution status is shown:

**Execution flow**: Components listed left-to-right in workflow order. Status color: completed (green) / running (blue) / waiting (gray).

```
1.rcp-setup ✓ → 2.impeller-agent ✓ → 3.diffuser-agent ✓ → 4.multi-agent ● → ... → 9.train (waiting)
```

**Component resource cards (3-column grid)**: Each component shows:
- Order number + name + type badge (component/solver/environment/train)
- Allocated GPU + memory usage (e.g., V100 x 1 · 12.3 / 16 GB)
- Usage progress bar (green: normal, red: >80%, blue: running)
- Status (✓ completed / ● running / waiting)
- Waiting components shown semi-transparent

9 components: rcp-setup → impeller-agent → diffuser-agent → multi-agent → make-blade → mesh-generator → run-cfx → environment → train

### 5.2 Per-Component Resource Monitoring (Real-time)

In reality, the workload runs as a single container (1 Pod, 1 image). However, the frontend must **display CPU/Memory/GPU allocation and usage per component as if each runs independently**.

#### Display per component

| Metric | Description | Format |
|--------|-------------|--------|
| CPU | Allocated vs used | Progress bar + value (e.g., 2.4 / 4 cores) |
| Memory | Allocated vs used | Progress bar + value (e.g., 12.3 / 16 GB) |
| GPU | Type/count + utilization | Progress bar + value (e.g., V100 x 1, 73%) |

#### Real-time variation

While the workload is running, each component's resource usage **changes over time**:
- Active components show fluctuating usage within a range typical for their type
- Completed components show 0% usage (resources released)
- Waiting components show "—" (not yet started)

Resource patterns by component type:

| Component Type | CPU | Memory | GPU |
|---------------|-----|--------|-----|
| component (setup, agent) | Medium (50~80%) | Low~Medium (30~60%) | Low (20~50%) |
| solver (blade, mesh, cfx) | High (80~95%) | Medium~High (60~80%) | High (70~90%) |
| environment | Medium (40~60%) | High (70~85%) | Medium (50~70%) |
| train | Medium~High (60~80%) | Very High (85~98%) | High (75~90%) |

#### Implementation

**Frontend simulation**: Resource values are simulated on the frontend based on component type patterns. Values fluctuate every 5 seconds within the pattern range. This is a demo-only approach.

| | Demo (current) | Production (future) |
|--|---------------|-------------------|
| Data source | Frontend simulation | Prometheus + DCGM |
| Refresh | 5s (setInterval) | 5s (API polling) |
| Per-component split | Type-based pattern simulation | Actual per-process measurement |
