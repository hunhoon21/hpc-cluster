# AVATAR OnE 플랫폼 기능명세서 변경사항 (v1.5 → v1.6)

## 변경 개요

v1.6은 **개발자 사전 테스트 및 결과 첨부** 기능을 도입하여, Trainer에서 학습 요청 제출 전에 본인이 직접 테스트를 실행하고 그 결과를 첨부할 수 있도록 합니다. 관리자는 첨부된 테스트 결과를 확인한 뒤 별도 테스트 없이 바로 승인할 수 있어, 승인 프로세스가 간소화됩니다.

---

## 1. Trainer 화면 변경

### 1.1 사전 테스트 실행 (신규)

**변경 전 (v1.5):** Trainer는 App 선택 → 리소스/파라미터 설정 → 요청 제출만 담당. 테스트는 승인 단계에서 관리자가 실행
**변경 후 (v1.6):** Trainer에서 학습 요청 제출 전 사전 테스트를 직접 실행 가능. 결과를 요청에 첨부하여 제출

| 기능 | 설명 |
|------|------|
| 사전 테스트 실행 | App·리소스·파라미터 설정 완료 후 "테스트 실행" 버튼으로 최소 실행 테스트를 직접 수행 |
| 테스트 결과 확인 | 테스트 완료 후 결과(로그, 성능 지표, 수렴 여부 등)를 Trainer 화면에서 바로 확인 |
| 결과 첨부 제출 | 테스트 결과를 학습 요청에 첨부하여 제출. "테스트 결과 첨부" 배지가 요청에 표시됨 |
| 첨부 없이 제출 | 사전 테스트 없이도 요청 제출 가능 (기존과 동일). 단, 관리자가 직접 테스트 필요 |

### 1.2 신규 기능 ID

| ID | 기능명 | 설명 | 우선순위 | 구현 단계 |
|----|--------|------|----------|-----------|
| TR-09 | 사전 테스트 실행 | Trainer 화면에서 학습 요청 제출 전 최소 실행 테스트를 직접 실행한다. RL 워크로드는 최소 에피소드 실행, 일반 워크로드는 최소 반복 실행 | P0 | Phase 1 |
| TR-10 | 테스트 결과 첨부 | 사전 테스트 통과 시 결과를 학습 요청에 첨부하여 제출한다. 첨부된 결과가 있으면 관리자가 별도 테스트 없이 확인 후 승인 가능 | P0 | Phase 1 |

### 1.3 TR-06 변경

**변경 전 (v1.5):** App + 리소스 + 학습 파라미터를 묶어 학습 요청을 제출한다
**변경 후 (v1.6):** App + 리소스 + 학습 파라미터를 묶어 학습 요청을 제출한다. 사전 테스트 결과가 있으면 함께 첨부된다

## 2. 테스트 및 승인 프로세스 변경

### 2.1 프로세스 개요 변경

**변경 전 (v1.5):**
> 승인이 필요한(needsApproval) 모든 워크로드에 대해 최소 실행 테스트를 수행한다. App 개발자가 테스트 결과를 학습 요청에 첨부할 수 있으며, 관리자도 자체적으로 테스트를 실행할 수 있다.

**변경 후 (v1.6):**
> Trainer에서 학습 요청을 제출하기 전, 개발자가 직접 사전 테스트를 실행하고 결과를 첨부할 수 있다. 개발자가 테스트 결과를 첨부한 경우, 관리자는 첨부된 결과를 확인하고 별도 테스트 없이 승인할 수 있다. 테스트 결과가 첨부되지 않은 경우에만 관리자가 직접 테스트를 실행한다.

### 2.2 승인 프로세스 흐름 변경

**v1.5 흐름:**
```
학습 요청 제출 → 관리자가 테스트 실행 → 결과 확인 → 승인/반려
```

**v1.6 흐름:**
```
[Trainer] 사전 테스트 실행(선택) → 결과 첨부하여 요청 제출
→ [관리자] 첨부 결과 확인 → 승인/반려
   또는
   테스트 미첨부 시 → 관리자가 직접 테스트 실행 → 결과 확인 → 승인/반려
```

### 2.3 AP-03 변경

**변경 전 (v1.5):**
> 최소 실행 테스트: 모든 needsApproval 워크로드에 대해 최소 실행 테스트를 수행한다.

**변경 후 (v1.6):**
> 최소 실행 테스트: 개발자가 사전 테스트 결과를 첨부하지 않은 needsApproval 워크로드에 대해 관리자가 최소 실행 테스트를 수행한다. 개발자 첨부 결과가 있는 경우 관리자 테스트 실행은 선택사항이다.

### 2.4 AP-05 변경

**변경 전 (v1.5):**
> 테스트 미실행 시 승인 불가 (경고 표시)

**변경 후 (v1.6):**
> 개발자 첨부 테스트 결과가 있으면 관리자가 결과를 확인(리뷰)한 후 바로 승인 가능. 테스트 결과가 없으면(개발자 미첨부 + 관리자 미실행) 승인 불가 (경고 표시)

### 2.5 AP-06 신규

| ID | 기능명 | 설명 | 우선순위 | 구현 단계 |
|----|--------|------|----------|-----------|
| AP-06 | 첨부 테스트 결과 리뷰 | 개발자가 첨부한 사전 테스트 결과를 관리자가 확인한다. 결과 로그, 성능 지표, 수렴 여부를 검토하고 "확인 완료" 처리한다 | P0 | Phase 1 |

## 3. 데이터 구조 변경

### 3.1 loopTest 데이터 구조 확장

| 필드 | 변경 | 설명 |
|------|------|------|
| executedBy | 값 확장 | `"developer"` \| `"admin"` — 테스트 실행 주체 |
| attachedToRequest | 신규 | `boolean` — 개발자가 요청 시 결과를 첨부했는지 여부 |
| reviewedBy | 신규 | 관리자 확인자 (첨부 결과 리뷰 시) |
| reviewedAt | 신규 | 관리자 확인 일시 |

### 3.2 데이터 예시 — 개발자 사전 테스트 첨부

```json
{
  "loopTest": {
    "status": "passed",
    "episodes": 10,
    "results": {
      "convergence": true,
      "avgReward": 85.3,
      "finalReward": 92.1,
      "log": "Episode 1: reward=72.1\nEpisode 2: reward=78.5\n..."
    },
    "executedBy": "developer",
    "executedAt": "2026-02-25 10:00",
    "attachedToRequest": true,
    "reviewedBy": "admin",
    "reviewedAt": "2026-02-25 11:30"
  }
}
```

### 3.3 데이터 예시 — 관리자 직접 테스트 (기존 방식)

```json
{
  "loopTest": {
    "status": "passed",
    "episodes": 10,
    "results": {
      "convergence": true,
      "avgReward": 85.3,

    "attachedToRequest": false,
    "reviewedBy": null,
    "reviewedAt": null
  }
}
```

## 4. 전체 워크플로우 변경 (섹션 1)

### 단계 2 변경

**변경 전 (v1.5):**
> 사용자가 Trainer 화면에서 개발 완료된 App을 선택하고, 리소스·학습 파라미터를 설정하여 학습 요청을 제출한다.

**변경 후 (v1.6):**
> 사용자가 Trainer 화면에서 개발 완료된 App을 선택하고, 리소스·학습 파라미터를 설정한다. 제출 전 사전 테스트를 실행하여 결과를 첨부할 수 있다. 테스트 결과 첨부 시 관리자 승인이 간소화된다.

### 단계 3 변경

**변경 전 (v1.5):**
> 승인이 필요한 모든 워크로드에 대해 최소 실행 테스트를 수행한다.

**변경 후 (v1.6):**
> 개발자가 사전 테스트 결과를 첨부한 경우 관리자는 결과를 확인하고 승인한다. 미첨부 시 관리자가 직접 테스트를 실행한 후 승인한다.
